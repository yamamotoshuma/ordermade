# 打者一巡対応の設計メモ

## 背景

現状の `batting_stats` テーブルは物理的には同一打者・同一イニングの複数行を保存できます。
しかしアプリ側が `同一試合 + 同一イニング + 同一打者 = 1件` という前提で衝突判定しているため、
打者一巡して同じ打者が同一イニングで再度打席に立った場合に新規登録できません。

野球の業務要件としてこれは不正です。
したがって、今回の対応では

- `batting_stats` を「1打席1行のログ」として扱い直す
- 既存データを壊さない
- 試合詳細など既存表示の大枠 HTML を崩さない
- 将来の代打・再入力・スクレイピングにも耐える

ことを優先します。

## 2026-04-24 追記: 表示要件が明確になった後の方針更新

表示について、

- 同一セルに複数打席を並べるのではなく
- `3回` の右隣にヘッダーなし列を追加して `3回の2打席目` を出す

という要件が明確になったため、採用案を更新します。

この表示要件に最も素直に対応できるのは、
`batting_stats` に **その打者のそのイニング何打席目か** を表す列を1つ追加する案です。

この列をここでは `inningTurn` と呼びます。

## 現状の問題点

### 1. 新規登録の衝突判定が野球のルールと矛盾している

`App\Services\BattingStatService::findExistingBattingStat()` は
`gameId + inning + userId/userName` で既存行を探し、
一致したら新規作成ではなく更新確認に入ります。

この仕様は「同じ打者が同じイニングで二度打席に立たない」という前提ですが、
実際の野球では打者一巡で普通に発生します。

### 2. 表示は「1イニング1セル」に複数打席を出せるが、順序と識別が弱い

`resources/views/game/show.blade.php` と `resources/views/batting/index.blade.php` は
各セル内で `battingStats` を全件ループしているため、
実は同一イニング内に複数打席があっても表示自体は可能です。

ただし次の課題があります。

- 取得順が明示されていない箇所があり、表示順が安定しない
- セル内での区切りが弱く、複数打席が読みにくい
- `batting.index` は既存成績があるセルだと `新規作成` 導線が消える

### 3. 成績と打順の紐付けが `userId / userName` ベースで弱い

現在の打撃成績表示は、打順行と成績行の対応を `userId` または `userName` で判定しています。

これは通常運用では大きく破綻しませんが、次のケースに弱いです。

- 同名選手
- 打順の入れ替え後に表示を見直したいケース
- 代打・途中出場の表現を将来強化したいケース

ただし `batting_orders` は保存時に `delete + insert` しているため、
`orderId` をそのまま `batting_stats` に外部キーで持つ設計は不向きです。

## 比較した案

### 案A: 衝突判定だけ外して、他はそのままにする

#### メリット

- DB変更なし
- 最短で打者一巡に対応できる

#### デメリット

- 表示が `userId / userName` 依存のままで将来の拡張に弱い
- 打順セルへの紐付けが曖昧なまま
- `batting.index` の `追加` 導線など別途UI調整が必要

### 案B: `batting_stats` を1打席ログとして扱い、`inningTurn` を追加する

#### メリット

- 既存テーブルを活かせる
- 既存データを壊さない
- `3回 / 3回(2打席目)` のような列展開にそのまま使える
- 1セル1打席になるので表示と編集導線が整理しやすい

#### デメリット

- nullable カラム追加と表示側のフォールバック設計が必要

### 案C: 別テーブルで全面的に打席イベントを作り直す

#### メリット

- 設計としては最もきれい

#### デメリット

- 既存の `batting_stats` 前提の画面・集計・テストへの影響が大きい
- 本番データ移行リスクが高い
- 今回の目的に対して過剰

## 採用案

**案Bを採用する。**

方針は次の通りです。

1. `batting_stats` は「1打席1行」として扱う
2. `同一打者・同一イニング` の衝突更新フローは廃止する
3. 代わりに二重送信・他端末競合は `offenseStateVersion` と名前付きロックで防ぐ
4. `batting_stats` に `inningTurn` を追加し、同一打者・同一イニングの1打席目/2打席目/3打席目を保持する
5. 表示画面では各イニングごとに `turn=1` を通常列、`turn=2..n` をヘッダーなし追加列として並べる

## 追加するカラム

`batting_stats` に以下のカラムを追加する。

- `inningTurn`

意味は以下です。

- 同一打者が同一イニングで何打席目か
- 1打席目は `1`
- 2打席目は `2`
- 3打席目は `3`

想定定義:

- `unsignedInteger`
- `default 1`

この形なら既存データは migration 時点で全件 `1打席目` として扱えます。
既存行の意味は壊れません。

## 保存時の扱い

### 新規登録

- 同一試合・同一イニング・同一打者の既存行数を数える
- `inningTurn = 既存件数 + 1` を採番する
- **既存行は探さず、常に insert する**

### 編集

- 編集で打者またはイニングを変更した場合は
  影響を受ける旧グループ / 新グループの `inningTurn` を振り直す
- 既存 row の `id` はそのまま使う

### 削除

- 削除後、同一試合・同一イニング・同一打者の残り行を `1, 2, 3...` へ詰め直す

`inningTurn` を持つ場合は、この再採番をやらないと
`1打席目が消えたのに2打席目だけ残る` という不自然な状態になるためです。

### 二重送信・競合

これまでの「同じ打者・同じイニングがあれば更新確認」はやめる。

代わりに以下で守る。

- `OffenseStateService::assertExpectedVersion()`
- 試合単位の named lock
- 送信中 UI ロック

これで accidental な重複登録は防げます。
そして、野球上正しい「同一打者・同一イニングの別打席」は新規登録できます。

## 既存データの移行方針

### 破壊的変更はしない

- 既存列は削除しない
- 新規列はすべて nullable で追加する

### バックフィル方針

今回のカラム追加は `default 1` で十分です。

既存データはすべて

- `inningTurn = 1`

として扱えば意味が壊れません。

すでに本番に存在する「1イニング内で同一打者が複数行あるデータ」がもしある場合だけ、
追加 migration または artisan command で `created_at, id` 順に `1, 2, 3...` を振り直します。

## 表示の変更方針

対象は主に以下です。

- `resources/views/game/show.blade.php`
- `resources/views/batting/index.blade.php`

### 表の構造は維持しつつ、列の持ち方を拡張する

スクレイピング互換性を考え、以下は変えない。

- inning ごとの `<td>`
- 打順ごとの `<tr>`

変えるのは「イニング列の定義」です。

例えば 3回に最大2打席目まで存在する場合、ヘッダーはこうなります。

- `1 | 2 | 3 | (空) | 4 | 5 ...`

ここで

- `3` が `inning=3, inningTurn=1`
- その右の空列が `inning=3, inningTurn=2`

を意味します。

3打席目まであるなら空列をさらに増やします。

### セル表示ルール

各セルは `1セル1打席` にします。

これにより

- セル内で文字列が連結しない
- 編集リンクを1件ずつ素直に置ける
- スクレイピング側も `列` として扱いやすい

という利点があります。

### `batting.index` の導線

既存成績があるセルでも、そのイニングの次 `inningTurn` 列へ追加できるよう
`追加` 導線を残すべきです。

具体的には

- 既存打席が入っているセルは編集リンク
- まだ存在しない次の `inningTurn` セルは `新規作成`

がよいです。

## Service 側の整備方針

Blade の三重ループが増えると可読性も性能も落ちるため、
表示用の grouped data は Service 側で作るのが望ましいです。

例えば以下のような keyed collection を作る。

- key: `playerKey:inning:inningTurn`
- value: そのセルに表示する `BattingStats[]`

これにより表示側は

- 打順行を回す
- `inning / inningTurn` 列定義を回す
- そのセルの単一打席を描画する

だけで済みます。

## 走塁状態管理との整合

今回の打者一巡対応では、走塁状態管理の根幹は大きく変えなくてよいです。

`inningTurn` は主に表示と編集導線のための列であり、
次打者解決そのものは従来通り時系列ログと攻撃状態から行えます。

## テスト方針

最低限、以下を追加・更新する。

### Feature Test

- 同一打者・同一イニングで2件目を新規登録できる
- 2件目登録時に `inningTurn = 2` になる
- 削除後に `inningTurn` が詰め直される
- 2件登録後、打者のデフォルトが次打者へ進む
- `batting.index` で 3回 の右隣に空ヘッダー列が追加される
- `game.show` で 3回 の右隣に空ヘッダー列が追加される
- 既存データは `inningTurn = 1` として従来通り表示される

### Visual Regression

- 打者一巡したゲームフィクスチャを追加
- `game.show` の打撃成績テーブルで同一セルに複数打席が入っても崩れないことを確認する

## 実装順

1. `batting_stats` に `inningTurn default 1` を追加
2. 既存データを必要に応じて安全にバックフィル
3. create/update/delete のたびに `inningTurn` を再採番する Service を用意する
4. 新規登録の衝突更新フローを廃止し、常に insert にする
5. 表示用 Service を整備して `game.show` / `batting.index` の列定義を動的化する
6. `新規作成` 導線を各 `inningTurn` 列へ整備する
7. Feature test / visual regression を更新

## 結論

今回の本質は「同一打者・同一イニングを一意とみなしていたアプリの前提を捨てる」ことです。

ただし全面再設計までは不要です。

**`batting_stats` はそのまま活かし、`inningTurn` を1列追加して、
新規登録を常に insert に戻す。**

これが、

- 既存データへの影響が最小
- 本番移行が安全
- `3回` の右隣に空列を足す表示要件に素直に合う
- 打者一巡・複数打席表示に耐える

という条件を最もバランスよく満たす実装です。
